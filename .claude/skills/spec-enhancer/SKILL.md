# SPEC 실현 가능성 검토 & 개선

SPEC 문서 하나만 보고 처리하지 않는다. 유저가 **무엇을 원하는지 먼저 고르게** 한 뒤,
- 🅰 **평가** — SPEC + 팀·시간·기술·보안 맥락을 모아 "3주(7/15) 안에 작동하는 작품을 완성할 수 있는가"를 판단하고 피드백
- 🅱 **개선** — 유저와 질의응답을 주고받으며 SPEC 자체를 더 명확하고 현실적으로 다듬기

## 입력
- 팀의 **SPEC 문서**(요구사항 초안). 유저가 붙여넣거나 파일로 제공한다.

## 시작 — 모드 선택 (무조건 먼저!)
SPEC을 받으면, **진행 전에 유저가 무엇을 원하는지 먼저 묻는다.** 선택하기 전에는 평가도 개선도 시작하지 않는다.

> "이 스펙으로 무엇을 해드릴까요?
> 🅰 **평가** — 이 팀이 3주 안에 작동하는 작품을 완성할 수 있는지 판단하고 피드백
> 🅱 **개선** — 스펙을 더 명확하고 현실적으로 함께 다듬기"

- 유저가 **둘 다** 원하면 **🅰 평가 → 🅱 개선** 순서로 진행한다.
- 모드는 중간에 바꿀 수 있다. (평가 끝에 개선을 제안하거나, 개선 중 완주 평가로 넘어가도 됨)

## 공통 — 먼저 SPEC 읽기
어느 모드든, 먼저 SPEC을 읽고 메모한다: 무엇을·왜·누구·핵심 기능·범위·이미 드러난 제약.
그리고 **SPEC에 이미 적힌 것은 다시 묻지 않는다.**

## 핵심 원칙 (먼저 읽기)
- **시작할 때 모드(평가/개선)부터 고르게 한다.** 선택 없이 바로 진행하지 않는다.
- **한 번에 질문을 쏟지 말 것.** 2~4개씩 묶어 대화로 주고받는다.
- **직접 묻기 어려운 것**(개인 역량 등)은 "잘하세요?"가 아니라 **행동·경험**을 묻는 구체 질문으로 우회한다.
- 평가·개선 모두 **줄 세우기가 아니라 완주를 돕기 위한 것.** 역량을 낮잡지 않고 건설적으로.
- 모르면 추측하지 말고 **질문**한다. 답이 모호하면 한 번 더 좁힌다.
- 모든 판단의 기준은 **"3주(7/15) · 이 팀의 현실"**.
- **개선은 한 번에 한 가지 약점만** 다룬다. 답을 받으면 **어떻게 반영할지 제안**하고, **동의를 받아** 고친다. 멋대로 다시 쓰지 않는다.

## 🅰 평가 모드
"이 팀이 3주 안에 완주 가능한가"를 팀·시간·기술·보안 맥락까지 모아 판단한다.

### A1 — 맥락 수집 (묶음 질문)
아래 차원을 순서대로, 묶음으로 묻는다. **각 차원에서 SPEC에 없는 것만.**
답에서 새로운 리스크가 보이면 그 자리에서 더 캐묻는다.

#### A. 팀 구성·역량  ← 직접 묻기 어려움, 그래서 우회 질문으로
- 팀원이 몇 명인가요?
- 그중 Claude Code로 기능을 **PR 머지까지 끝내본** 사람은 몇 명인가요?
- **git 브랜치·PR·충돌 해결**이 익숙한 분이 몇 명인가요?
- 지금까지 만든 것 중 **가장 복잡했던 프로젝트**는 무엇이었나요?
- 막혔을 때 주로 어떻게 푸나요? (공식 문서 / 검색 / AI에게 물어봄 / 동료)
> "실력이 좋냐"고 직접 묻지 말 것. 위처럼 **경험·행동**으로 가늠한다.

#### B. 가용 시간
- 3주 동안 **하루 평균 몇 시간** 이 프로젝트에 쓸 수 있나요?
- 본업·다른 일정과 **병행**인가요, 집중할 수 있는 기간이 있나요?
- 팀원들이 **같은 시간대**에 일하나요(동기) 아니면 각자 시간에(비동기)?

#### C. 기술·환경·보안 제약  ← 구체적으로
- 개발 환경에서 **외부 인터넷·외부 API 호출**이 되나요? (사내망 차단 여부)
- 필요한 **패키지·도구를 설치할 권한**이 있나요?
- 다룰 **데이터에 민감정보**(고객·금융·개인정보)가 있나요? **더미 데이터**로 대체 가능한가요?
- 결과물은 **어디서 시연**하나요? (개인 PC 로컬 / 사내 서버 / 외부 배포)
- 반드시 써야 하는, 또는 쓰면 안 되는 **기술 스택·사내 표준**이 있나요?

#### D. 기술 스택 친숙도·외부 의존성
- SPEC의 기능을 만들려면 팀이 **안 써본 기술**이 있나요? (학습 곡선)
- **외부 서비스·외부 데이터**에 의존하나요? (그게 막히면 작품도 막힌다)

#### E. 범위·리스크  ← 마무리 질문
- 이 중 **딱 하나만 된다면** 무엇이 가장 중요한가요? (핵심 1개)
- 지금 가장 **불안하거나 불확실한** 부분은 무엇인가요?
- 7/15 **발표 때 보여줄 한 장면**을 상상하면 무엇인가요?

> 위 5개 차원은 출발점이다. 팀·SPEC 성격에 따라 차원을 더하거나 뺀다.

### A2 — 종합 평가
모은 정보를 합쳐 **3주 완주 가능성**을 중심으로 판단한다. 보는 관점:
- **범위 ↔ 역량 ↔ 시간의 균형** — 핵심 기능의 크기가 (팀 역량 × 가용 시간) 안에 들어오나?
- **치명적 제약** — 외부 API 차단 · 설치 권한 없음 · 민감정보 등 **작품을 막을 요인**이 있나?
- **가장 큰 단일 리스크** — 무엇이 깨지면 7/15에 작동하는 작품이 안 나오나?
- **학습 곡선** — 안 써본 기술이 3주를 통째로 잡아먹지 않나?

### A3 — 피드백 (두괄식)
**결론부터** 짧게 3블록으로 전달한다. (장황한 보고서 금지)
1. **종합평** — 두괄식 한 줄 결론: 🟢 지금 범위로 완주 가능 / 🟡 범위를 줄이면 가능 / 🔴 지금 범위로는 위험. + 핵심 메시지 한 줄.
2. **잘한 점** — 1~3개 (살릴 강점)
3. **부족한 점** — 1~3개 (완주를 위협하는 것 중심: 범위·모호함·리스크)

그다음 **세부 피드백을 원하는지 묻는다** (확인 없이 쏟아내지 않는다):
> "여기서 더 깊이 볼까요? 원하시면 어느 부분을 자세히 짚어드릴지 골라주세요 — **범위 / 핵심 기능 / 일정·시간 / 기술·보안 / 리스크** 중에서."
- **원하면** → 고른 부분에 대해 **추가로 필요한 정보를 질문**하고, **구체적인 세부 피드백·조정 제안**을 준다. (한 번에 하나씩)
- **원치 않으면** → 🅱 **개선 모드**로 함께 다듬거나, "`grill-me`/`/to-prd`로 넘기면 됨"으로 마무리한다.

## 🅱 개선 모드
스펙 자체를 질의응답으로 고쳐 더 명확하고 현실적으로 만든다.
(팀 맥락까지 캐물을 필요는 없지만, **범위가 3주에 맞는지**는 본다)

### B1 — 약점 스캔
SPEC을 훑어 빈틈을 찾고, 무엇이 약한지 유저에게 **짧게 공유**한다.
우선순위: **범위 → 모호한 문제 → 빠진 핵심**. 겨냥하는 것:
- 범위가 큼 → 핵심 1~2개만 남기고 나머지는 **'안 할 것'**으로
- 문제가 모호 → **누가·언제·왜** 불편한지 구체화
- 핵심 기능이 화면·솔루션으로 적힘 → **'무엇을·왜'**로 되돌리기
- 성공 모습이 없음 → **"뭐가 되면 성공"**인지 한 줄
- 기술 명세(How)가 섞임 → **구현 단계로 빼기**
- 열린 질문이 남음 → **하나씩 결정**

### B2 — 질의응답 티키타카로 개선
약점을 **한 번에 하나씩** 고쳐나간다:
1. 약점 하나를 짚고, 좁히는 **질문 1~2개**를 던진다.
2. 유저의 답을 받는다.
3. 그 답을 스펙에 **어떻게 바꿀지 구체적으로 제안**한다. (예: "핵심 기능을 5개→2개로 줄이고, 나머지는 '안 할 것'으로 빼죠")
4. 유저가 동의하거나 수정하면 **반영**한다.
5. 다음 약점으로 넘어간다.

- 한 바퀴 돌 때마다 **지금까지 바뀐 핵심을 한두 줄**로 짚어 누적을 보여준다.
- 약점이 해소되거나 유저가 **"충분하다"**고 하면 멈춘다.

### B3 — 개선된 SPEC 산출
- **업데이트된 SPEC 전문**을 마크다운으로 출력한다. (유저가 그대로 복사·저장할 수 있게)
- 맨 아래 **핵심 변경 3~5줄 요약**을 붙인다.
- 마무리 한 줄: "이제 이 스펙으로 **`grill-me`**(더 깊은 정리) 또는 바로 **`/to-prd`**로 넘기면 됨."

## 출력 길이
- 모드 선택·질문·개선 대화는 **짧게 주고받는다.** 평가·피드백은 **한 화면 분량**으로 압축한다.
- 단, **B3의 개선된 SPEC 전문은 전체를 출력**한다(복사용이므로 줄이지 않는다).

## 하지 말 것
- **모드(평가/개선) 선택 없이** 바로 시작하기
- 질문을 한꺼번에 10개씩 던지기 (대화가 막힌다)
- SPEC에 이미 적힌 것 다시 묻기
- 역량을 "잘하냐/못하냐"로 직접 묻거나 단정하기
- 정보가 부족한데 추측으로 평가 내리기
- 평가 피드백을 **두괄식 요약(종합평 먼저) 없이** 길게 늘어놓거나, **세부 피드백을 유저 확인 없이** 쏟기
- 기술 구현 방법(How)을 평가하려 들기 — 여기선 **완주 가능성**과 **스펙 명료성**만 본다
- 개선할 때 약점을 **한꺼번에 다 고치려** 들기 (반드시 한 번에 하나씩)
- 유저 **동의 없이** 스펙을 멋대로 다시 쓰기
- 유저가 평가만 원하는데 **개선을 강요**하기
