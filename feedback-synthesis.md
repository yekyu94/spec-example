🟢 **지금 범위로 완주 가능 — 받은 스펙 중 가장 앞서 있는 팀.**
시스템은 이미 돌고, 유일하게 남은 LLM 연동도 "API 키 발급(오늘 예정)" 수준으로 좁혀졌다.
이 팀의 과제는 **"더 만들기"가 아니라 "안 깨지게, 잘 보여주기"**다. 두 피드백이 이 점에서 완전히 일치한다.

## 잘한 점 (두 피드백 종합)

1. **이미 작동하는 작품이 있다 — 최대 강점.** 크롤→사이트맵→건강성 테스트→수동 보정→결과까지 비-AI 경로가 다 돈다. 해커톤 최다 실패("발표날 안 켜짐")를 이미 통과. *(양쪽 공통)*
2. **시연 안정성을 구조에 내장했다.** 키가 없어도 건강성 점검은 항상 동작하는 graceful degradation + mock 사이트(`:10002`/`:10003`)로 "키 없이 전체 흐름 시연" 가능. 키·네트워크 문제로 데모가 멈출 위험을 스스로 차단. *(양쪽 공통, 다른 팀이 참고할 설계)*
3. **문제→해결 흐름이 군더더기 없다.** ① URL 등록 → ② 크롤·트리 → ③ 자동/AI 테스트 → ④ 운영↔개발 AI 비교. 무엇을·왜가 한눈에. *(original)*
4. **데이터 단일 소스 원칙이 분명하다.** `(environment_id, url)` 키의 `TestResult`를 사이트맵/결과/리포트가 공유 → 기능이 늘어도 데이터가 흩어지지 않음. *(original)*
5. **스펙이 구현과 정합적이며 자기점검 수준이 높다.** 데이터 모델·API·status enum이 코드 기준으로 기술됨. Non-goals(인증·CI 연동 없음)와 §9 리스크(0.0.0.0+무인증)까지 스스로 닫음. *(양쪽)*

## 개선/주의할 점 (완주·전달 중심)

1. **초점을 '추가'가 아니라 '안정화·전달'에.** 이 문서는 요구사항 초안이 아니라 사실상 동작하는 시스템의 명세서다. 이 단계 최대 함정은 "시간 남으니 기능 더 넣자" → 데모가 깨짐. 평가 비중이 큰 **완성도(끝까지 도는가)와 발표(문제→해결 전달)**에 남은 시간을 쓸 것. *(original)*
2. **"키만 꽂으면 끝"은 낙관일 수 있다.** 키 연동과 **AI 출력 품질**(프롬프트·`_parse_json` 안정성·비용/속도)은 별개 작업. 키 꽂은 뒤 *실제 평가 결과가 쓸 만한지* 검증할 시간을 따로 잡을 것. *(feedback)*
3. **데모 경로 하나로 고정 + 미해결 질문 닫기.** §11의 `compare` vs `compare-run` 단일화, README 포트 불일치(8000/5173 vs 10000/10001)는 시연 직전 발목을 잡기 쉽다. 발표용 탈 시나리오 하나를 정하고 그 경로·포트만 확실히 정리. *(양쪽 공통)*
4. **사용자(Who)가 얇다.** 문제·기능은 강한데 사용자가 "QA 담당자"로만 서술됨. *어떤 QA가 지금 어떤 환경에서 어떻게 일하는지* 한두 줄만 더하면 "이 사람의 이 고생이 이렇게 줄었다"가 선명해져 발표 전달력↑. *(original)*

## 두 피드백의 합의 / 보완 관계

- **합의 (신뢰도 높음):** ① 이미 도는 시스템이 최대 강점 ② graceful degradation·mock = 데모 안정성 ③ 과제는 "안 깨지게, 잘 보여주기" ④ 미해결 질문(compare 단일화·포트)을 닫아라.
- **`original`의 고유 기여:** 데이터 단일 소스·Non-goals 자기점검 강점, 사용자(Who) 구체화, "안정화·전달로 초점 이동" 프레이밍.
- **`feedback`의 고유 기여:** 팀·시간·네트워크·키 발급 등 실제 맥락 확보, "키 연동 ≠ AI 품질 완성" 경계, 데모 재현성.
